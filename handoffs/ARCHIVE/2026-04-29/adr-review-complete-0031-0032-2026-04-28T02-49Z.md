## Review: ADR-0031 — Explicit layer separation for context, canon, knowledge, and memory

**Reviewer**: supervisor-tick (claude/sonnet-4-6)  
**Date**: 2026-04-28T02:49Z  
**Verdict**: Sound — no blocking findings

### What the ADR claims

A four-layer stack: (1) agent context repos, (2) canon, (3) knowledge system, (4) memory/retrieval surfaces. Each layer is distinct; cross-layer comparisons are category errors unless framed explicitly. `context-repository` co-locates layers 1 and 2 by design but is not layers 3 or 4.

### Pressure tests

**1. Is the four-layer split stable when vendors combine layers?**

Challenge: Tools like Graphiti can hold both canonized evidence (layer 2 content projected) and runtime retrieval (layer 4). Does the layer model survive vendor combination?

Assessment: The ADR accounts for this — retrieval surfaces "may read from or write into" canon-backed projections. The key invariant is that retrieval surfaces are not *authoritative by themselves*. Defensible; the burden is on system code to enforce the authority constraint, not just name it.

**2. Is "knowledge system" concrete enough to be actionable?**

Challenge: Layer 3 is defined as "validated invariants and reusable methodologies that persist across pod lifecycles." IDEA-0005 owns the physical-home decision. Without a physical home, is this layer aspirational scaffolding?

Assessment: Legitimate concern. The ADR correctly scopes IDEA-0005 as the next decision point and doesn't claim layer 3 is implemented. Risk: the layer remains aspirational and the vocabulary becomes load-bearing without substance. **Watch item: IDEA-0005 must close before layer-3 vocabulary is used to justify architectural moves.**

**3. Does co-locating layers 1 and 2 in `context-repository` create confusion?**

Challenge: New sessions will read `context-repository` and assume it *is* canon, not that it *hosts* canon.

Assessment: Moderate risk. The mitigation (requiring front-door docs to use four-layer vocabulary) is correct but depends on future compliance. A stronger mitigation would be physical namespace separation (e.g. `spec/` = canon, `docs/` = context-repo pattern). The ADR's follow-on #1 calls for this but doesn't mandate physical separation. Acceptable for now; track compliance in reflections.

**4. Does this ADR conflict with accepted ADRs?**

ADR-0027 (synaplex reframe): additive, not contradictory. ADR-0027 defines product topology; ADR-0031 defines epistemic layer stack. They compose cleanly.

### Verdict

Sound. No supersession required. Watch items: (a) close IDEA-0005 before layer-3 vocabulary drives architecture, (b) front-door docs must adopt four-layer vocabulary explicitly.

## Review: ADR-0032 — Cowork is an external friction surface and secondary to the system backlog

**Reviewer**: supervisor-tick (claude/sonnet-4-6)  
**Date**: 2026-04-28T02:49Z  
**Verdict**: Sound — no blocking findings

### What the ADR claims

Cowork is secondary by default: it yields to executive INBOX triage, synthesis follow-through, active project backlog, and structural blockers. It may not become a validation gate, required reviewer, or prerequisite for dispatch or closure.

### Pressure tests

**1. Is the "secondary by default" rule self-enforcing?**

Challenge: The rule says cowork yields, but if cowork emits something urgent-sounding, there's no mechanical gate preventing the executive from treating it as primary. The rule is honor-system.

Assessment: Accurate. The ADR's escalation clause (point 5: >1 new supervisory items without collapsing existing ones → tighten scope) is the closest thing to a mechanical gate, but it's still judgment-based. This is acceptable for a friction surface that has no write authority. The real guard is that cowork can only write to `runtime/friction/cowork/` and has no INBOX or project-code access.

**2. Does the ADR handle cowork-originated FRs correctly?**

Challenge: Cowork may identify real structural issues. If those are always "secondary," do they get delayed too long?

Assessment: The ADR correctly says cowork "may sharpen judgment" — the content can be promoted. The secondary-priority rule governs when the *reading* happens (after core triage), not whether the content is valid. A real structural finding from cowork should be promoted to an FR and enters the normal friction queue. This is not a problem.

**3. Does "silence is not degradation" create a monitoring blind spot?**

Challenge: If cowork goes silent for a cycle and that's "acceptable," how does the system detect a broken cowork connection vs. genuine silence?

Assessment: Low risk. Cowork is explicitly non-load-bearing. A broken connection degrades a secondary friction input, not a control-plane dependency. The ADR is correct that absence must not block execution. If cowork becomes load-bearing in the future, this ADR would need revisiting — but that would also require a new ADR explicitly making it load-bearing.

**4. Does this ADR conflict with accepted ADRs?**

No conflicts found. ADR-0032 is consistent with the action-default contract (ADR-0020) and the executive boundary discipline (FR-0018).

### Verdict

Sound. No supersession required. The honor-system nature of the secondary-priority rule is the main weakness, but it's acceptable given cowork's constrained write authority.
