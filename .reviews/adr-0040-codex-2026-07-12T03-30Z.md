## Reviewer 1 findings, after Codex's attack

### F1 — Policy is schema-forced and was wrongly excluded. **SURVIVES (blocking).**

ADR-0040 §In scope: "Emitters for the four envelope types an eval actually produces:
Claim, Evidence, Decision, EventLogEntry. **Not Policy**…"

But `decision.schema.json` requires `policies_in_force` with `minItems: 1`, and
`canon.md` validator rule 5 requires the referenced Policy version to exist. Canon
obligation 7 ("What policy applied at the time of each Decision?") is the reason.
There is no legal eval-terminating Decision with zero Policy envelopes. Atlas hit this
and emits `atlas.evidence_quality_to_canon_tier`, cited by every Decision it writes.

Codex escalated this correctly: it does not patch, it **invalidates the ADR's stated
scope**. `policy.schema.json` requires `class, scope, field_path, value, version,
issuer, amendment_authority, ratification_rule, rollback_rule, provenance,
effective_from` — Policy is a governance object, not a support record. "Add Policy,
minimally" is not a minimal amendment.

### F1b — the pre-registration itself is in conflict with canon. **NEW — the sharpest finding in the review.**

Neither ADR-0040 nor reviewer 1 caught this; Codex did.

`lab/evals/memory-systems-v1/methodology.md` §Related canon objects, inside the
**hash-bound, immutable, pre-registered** artifact:

> Policy (eval promotion-gate): declared inline in the Claim's `thresholds`;
> **no separate Policy object for v1**

So the eval, *as pre-registered*, declares it will not emit the object that canon
requires its terminal Decision to cite. This is not an emitter scoping oversight. The
eval cannot conclude in a schema-valid Decision no matter what the emitter does, and
the artifact cannot be edited without destroying the pre-registration it documents.

This is a third structural blocker on `memory-systems-v1`, alongside the missing
emitter and the missing runner. It must be resolved by a decision, not discovered by
an implementer at emission time.

### F2 — "the two emitters use different id contracts" is factually false. **FACT SURVIVES; PRESCRIPTION DOES NOT.**

ADR-0040 open question 2 asserts atlas and synaplex "use a *different* id contract"
and recommends keeping them separate and documenting the divergence.

Verified: atlas's `claim_hash()` = `sha256(claim_canonical(statement))[:16]`, where
`claim_canonical` = strip → lower → collapse whitespace → strip trailing `[.!?,]`.
Run against the synaplex Claim's statement it returns **`b7ff216f4eec6e58`** — the
existing envelope's id, exactly. The contracts are the same hash family; atlas's is a
strict refinement in normalization. The ADR's premise is wrong, so open question 2 was
about to be settled on a false basis.

Reviewer 1 concluded "converge on atlas's canonicalization." Codex refuted this and is
right: atlas's helper is a repo-local utility, convergence means either vendoring a
second copy across a repo boundary with no shared test surface (a new drift hazard —
the exact class S1-P3 names) or an L2 extraction that ADR-0029 and ADR-0040 both defer
until synaplex has run several evals. Codex also notes that in a *pre-registration*
store, wider normalization widens the equivalence class before the emitter can preserve
the distinction.

Resolution carried into ADR-0041: pin `sha256(statement.lower())[:16]` (ADR-0040's own
recommendation (a)), and record the verified fact that atlas's stricter derivation
reproduces the same id — i.e. the contracts are *compatible today*, and the divergence
is in normalization strictness, not hash family. That is the true statement, and it
keeps L2 extraction cheap later without forcing a cross-repo copy now.

### F3 — record the methodology id-drift in canon via `methodology_log`. **WITHDRAWN.**

Reviewer 1 proposed emitting the `memory-systems-v1-h1` → `b7ff216f4eec6e58`
reconciliation in the probe-entry `EventLogEntry(methodology_log).summary`.

Codex refuted it and is right. `summary` is optional free-form narrative; canon.md §9
reserves structured fields for exactly what is replay- or audit-load-bearing. Putting
an id mapping there makes audit depend on prose parsing, and it smuggles a canon v0.1.0
gap past the L1 freeze rather than escalating it.

The correct read is simpler than either proposal: **this is not a canon violation at
all.** Validator rule 7 requires `ArtifactPointer.content_hash` to reproduce from the
artifact — it does. The stale `memory-systems-v1-h1` label is prose *inside* the
artifact, not a canon reference. Canon is self-consistent. The drift is a documentation
defect in an immutable file, already recorded in `CURRENT_STATE.md` and project
`CLAUDE.md`. No canon envelope is needed, and none should be emitted.

ADR-0041 records this explicitly so a future session neither "repairs" the artifact
(destroying the pre-registration) nor manufactures a reconciliation envelope.

## Codex objections not folded into F1–F3

**Anti-kernel guard is prose, not mechanism (blocking).** ADR-0040's only mechanical
guard against reviving the reverted `lab/campaign` kernel is acceptance criterion 7,
"No file under `lab/campaign/` exists." That binds a *path*, not a *behavior* — the
same kernel at `lab/pressure/` satisfies it. Carried into ADR-0041 as a behavioral
invariant: the emitter serializes, validates, and writes; it contains no code that
*selects* what to emit.

**The four ADR-0038 constraints are not "all mechanically enforceable" (serious).**
ADR-0040 §ADR-0038 alignment claims they are. ADR-0038 §Reference direction says the
opposite in terms: "The grep/path guard can catch path citation laundering. It cannot
catch copied content. The copy/re-ingest ban is therefore a reflection-review
obligation, not a fully mechanical guarantee." An emitter that hash-pins an artifact
file will happily pin Programme prose copied into that artifact — no
`reasoning/programmes/` path ever appears. Three of the four constraints are
mechanical; the fourth is not, and the ADR must not advertise a guarantee it does not
provide.

**"The blocker is that there is nothing to write with" is incomplete (serious).**
ADR-0040 §Context frames the missing emitter as *the* blocker. The eval runner also
does not exist (0 python files under `lab/`). The emitter is necessary, not sufficient.
Naming one blocker as "the" blocker is the same overclaim that let this sit for 12
weeks.

**Codex: "no runner ⇒ premature."** Rejected. The near-term consumers are real and
precede the runner: the probe-entry `EventLogEntry` + `methodology_log` that canon
requires on probe entry, and the control-arm rival Claim whose pre-registration window
is only open while zero Evidence exists. Hand-authoring those is what produced the drift
being cleaned up. But Codex is right that this does not make the work *urgent*, and
nothing here justifies rushing the rival Claim ahead of a reviewed emitter.

## Verdict

**Reject ADR-0040.** The direction is endorsed and most of the analysis is sound, but
the proposal's central scope claim is provably false, one supporting premise is
factually wrong, one guarantee is overclaimed, and its anti-kernel guard binds a path
instead of a behavior. Accepting it — even with amendments pencilled in the review —
would put a decision of record on the shelf whose reasoning does not support its
conclusion, and would authorize an emitter that cannot complete the one eval it exists
to unblock.

Superseded by **ADR-0041**, which carries the endorsed direction forward with the
corrected scope and names the pre-registration/canon Policy conflict as a first-class
problem to be decided rather than discovered.

## Round 2 — ADR-0041 (the executive's own proposal, rejected)

ADR-0041 fixed 0040's scope error by bringing Policy in scope, then made a worse error:
it mandated Policy governance semantics **canon cannot express**.

- `policy.md` §Classes gives exactly two classes. **`operational`** = "Agent-mutable within
  constitutional ceilings," amendable by `Decision(kind=amend_policy)` citing Evidence — a
  gate an agent can move *after seeing the results*. **`constitutional`** = principal-only,
  framework-level (capital ceilings, kill-switch, amendment authority itself). A
  pre-registered eval gate is neither.
- ADR-0041 tried to close the hole by requiring `amendment_authority` to forbid in-eval
  loosening. `policy.schema.json` types that field as a bare array of principals, and
  `ratification_rule.kind` ∈ `{principal_signoff, evidence_threshold, multi_party}`. The
  constraint is not merely unenforced — it is **unwritable**.
- Pinning `value` alone does not give "zero new degrees of freedom": `scope`, `field_path`,
  `ratification_rule`, and `rollback_rule` stay discretionary. Codex's sharpest example — a
  `ratification_rule: evidence_threshold` with count `0`, a gate ratified by no evidence.
- "Byte-identical `value`" would have shipped a broken test: the Claim's thresholds contain
  the lexeme `0.80`, which round-trips to `0.8` through any normal serializer.

**The finding that decided everything downstream:** the real canon gap is not
`policies_in_force: minItems 1` (that is canon working as designed — obligation 7). It is that
**canon v0.1.0 cannot encode a frozen, pre-registered, eval-local promotion gate.** Project
`CLAUDE.md` requires canon gaps to escalate to context-repository rather than get a local
workaround. ADR-0041 cited that rule and then built the workaround.

## Round 3 — ADR-0042 (accepted)

ADR-0042 takes round 2's conclusion literally: it **splits the emission path at the Policy
boundary**. Only `Decision` has a Policy dependency, and it is the last envelope an eval emits
— so Phase 1 (Claim, Evidence, EventLogEntry) proceeds, Phase 2 (Decision, Policy) blocks on
the escalated canon gap.

Codex verdict — **ACCEPT-WITH-AMENDMENTS**:

- **Q1 (decisive): the split is sound.** No Policy dependency in Claim, Evidence, or
  EventLogEntry — not in `required`, not in obligations, not in phase invariants, not in
  validator rules. Entering `probe` requires a methodology-log EventLogEntry and
  pre-registration immutability; it triggers no Policy obligation. Independently verified by
  the executive against `canon.md` §Phase invariants before the review returned.
- **Q2: an eval stuck mid-probe is incomplete, not a violation.** Obligation 3 asks what
  Decision was made; mid-probe, the answer is "none yet." `canon.md:5` — "an obligations model,
  not a state machine" — saves it. *It becomes a violation the moment the eval is represented as
  concluded or published as findings without a valid Decision.*
- **Q3: the live hazard**, found independently by both reviewers. A store of Claims + Evidence
  with zero Decisions lets Layer 4 publish un-concluded results. Project `CLAUDE.md` gates
  publication on "the matching canon envelope passing validation" — and a valid *Evidence*
  envelope satisfies that reading. The Claim's own second falsification criterion pre-registers
  that the held-out suite may reverse the ordering and that the Decision *must* note the
  override — so publishing Evidence pre-Decision would publish exactly the finding the Decision
  exists to override.
- **Q4:** validator rules 1 and 7 are the right Phase-1 subset; no rule missed.
- **Q5:** `memory-systems-v1` is **not void** and must not be abandoned or silently "rescued"
  by editing the methodology. It is preserved as pre-registered, with the gap recorded and
  conclusion blocked until Phase 2.

### Amendments applied to ADR-0042 before acceptance

1. **Layer 4 publication guard** (AC 9, fail-closed in `python -m integrity`): no reader-facing
   surface may publish a result, finding, conclusion, or promotion for a Claim without a
   validator-passing `Decision` citing that Claim and its Evidence.
2. **Pre-registration pages may remain public only while they publish no results.** The existing
   `site/src/pages/lab/memory-systems-v1.astro` qualifies and must not be "updated with early
   numbers."
3. **`memory-systems-v1` is recorded as `incomplete`, not `concluded`,** until a
   validator-passing Decision exists.

Because Phase 2 is blocked, no Decision can exist — so amendment 1 currently forbids every
results page. That is the intended behavior, and it is fail-closed by construction: the guard
requires an object that cannot yet be built, so it needs no judgment call to hold.
