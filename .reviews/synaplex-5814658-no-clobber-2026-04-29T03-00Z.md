# Adversarial Review — synaplex 5814658

## Challenges (from reviewer)

1. **`merge_jsonl_by_id` is not durable across a crash mid-write.** No
   `fp.flush() + os.fsync(fp.fileno())` on the tmp before close, and no
   `fsync` on the parent directory after the rename. POSIX `rename` is
   atomic for *visibility* but not for *durability*: a power loss
   between `os.replace` and the next periodic flush can leave a
   zero-byte file at the target path while the old inode is gone — exactly
   the symptom this commit was written to prevent.

2. **`.tmp` is leaked on any exception inside the `with open(tmp, "w")`
   block.** If `json.dumps` raises, or `ENOSPC` hits mid-write, the tmp
   remains on disk and `os.replace` is never called. Existing file
   survives (good), but no try/finally → orphan tmps accumulate.

3. **`raw_path()` keys by source only, not by (source, beat).** All
   beats sharing a source write to the same daily file. With merge-by-id
   semantics, beat A's items can be silently *replaced* by beat B's
   items if `content_id` collides — `latest wins` strips the `beat`
   field of the earlier writer. Latent today (one beat); becomes data
   corruption at 2nd beat.

4. **No file lock; concurrent writers race the read-modify-write
   window.** Manual smoke + cron coinciding causes the second writer's
   union to win, dropping the first writer's net-new items without
   trace.

5. **Reader sees half-written tmp via directory scan.** The score job
   opens by exact filename, so currently safe; any future glob over
   `raw/` would pick up `.jsonl.tmp` files.

6. **Day boundary race.** `date` resolved at function entry; if a fetch
   straddles midnight, the next 4h cron run's items collide with
   yesterday's entries by `content_id`, and "latest wins" mutates
   yesterday's count downward.

7. **`_gather_week` "highest score wins" amplifies rubric drift.** If
   the heuristic scorer changes mid-week, day-7's v2 score and day-1's
   v1 score are incomparable; current policy silently picks whichever
   is numerically larger. Should pick by `scored_at` (newest rubric)
   or detect drift.

8. **`emit_failure` ref points at a path that doesn't carry the data
   the failure was about** on early-return error paths. Minor triage
   hygiene.

## Verdict (from reviewer, verbatim)

> Reconsider before next adapter is added. The atomicity story is
> strong against process crashes but weak against host crashes,
> concurrent writers, day-boundary races, and the `(source, date)`-only
> path key. Ship-as-is is acceptable *only* while there is exactly one
> beat and one writer per source; the second beat or a manual smoke
> alongside cron will produce silent data loss. Minimum bar before
> declaring this verified: add `flock` + `fsync` + `try/finally`, fix
> `_gather_week` tiebreak, and document the single-beat-per-source
> assumption in `paths.py` until `raw_path` is re-keyed.

## Disposition (this session)

**Fixed in follow-up commit (low-risk, mechanical):**
- §1 fsync on file + parent directory after atomic rename
- §2 try/finally around the tmp write with cleanup-on-exception
- §8 friction-event `ref` on early-failure paths uses empty-string
  when path doesn't reflect the failure target

**Carried forward (larger design changes; tracked in CURRENT_STATE.md):**
- §3 `raw_path` keying by (source, beat) — latent until 2nd beat;
  documented as a structural followup. The `(source, date)`-only
  assumption is explicitly noted in `paths.py` per reviewer's minimum
  bar.
- §4 `flock` for concurrent writers — current cron-only operation has
  near-zero contention; manual+cron interleave is rare. Documented as
  a known sharp edge to address before any second writer is added.
- §6 day-boundary race — affects the cron firing closest to midnight
  (at our spacing, 00:17 UTC). Bound this by passing the date as a
  parameter computed at fetch-START (not at adapter-entry) when the
  fetch could span minutes. Carried forward.
- §7 `_gather_week` rubric-drift tiebreak — change to `scored_at`
  newest. Requires plumbing `scored_at` into the scored item shape;
  carried forward, current heuristic-only mode makes it low-priority.

The reviewer's "minimum bar before declaring this verified" is now
partially met: fsync + try/finally land this turn. flock and the path
re-key are scoped follow-ups; documented as such in the project's
Known broken section.
