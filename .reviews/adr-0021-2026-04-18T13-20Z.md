1. **Most dangerous assumption**

The ADR assumes the Claude hook surface can both fire reliably at session init and inject file contents into model context in a way that is equivalent to enforced reading (`lines 34-37, 41-43, 51-60, 125-128`). If that assumption is wrong, you create workspace-wide false confidence in “structural enforcement” while actually preserving the original failure mode under a more authoritative-looking mechanism.

2. **Missing failure mode**

It does not account for stale-context amplification: read enforcement is being added before freshness enforcement, while session-end updates are explicitly deferred (`lines 62-67`). In practice that means outdated `CURRENT_STATE.md` content will be auto-injected and gain more trust, not less; the design treats “was loaded” as if it were close to “was current” (`lines 73-77`), which is exactly the kind of failure that will recur.

3. **Boundary most likely to be collapsed in practice**

The repo-local declaration vs host-local enforcement boundary will erode first. The design presents `context-always-load:` in project `CLAUDE.md` as if it makes domains “self-activating” (`lines 75-77, 129-130`), but the actual mechanism lives in per-host dotfiles and is explicitly Claude-only for now (`lines 41-43, 88-90, 94-99`). In real use, people will treat the repo declaration as universal policy and forget that portability, Codex parity, and subagent behavior all still depend on out-of-repo setup.

Overall verdict: the ADR is directionally right about discipline-only policy being insufficient, but it overstates what this hook would prove even if implemented. As written, it risks replacing one silent failure mode with a broader false-assurance failure mode.
