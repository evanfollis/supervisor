## Policy refinement loop

Treat executive/supervisor operation as an explicit policy-search loop:

- friction is the training signal
- charter, playbooks, prompts, and current-state surfaces are the policy
- repeated successful outcomes are evidence the policy improved
- repeated nudges, hesitation, or corrections mean the policy is still wrong

The point is not to complete isolated tasks more cleanly. The point is to use
observed friction to refine policy so the stack needs less correction over
time.

The boundary is load-bearing: if the executive/supervisor stack becomes "just

- **Next action**: the single most important thing the next instance should do
- **Artifacts**: paths to any files the next instance needs to read

Keep it under 100 lines. If longer, promote content to `decisions/` or `playbooks/` and reference by path.

## Cross-agent handoff (claude ↔ codex)

Handoff files are agent-agnostic (plain markdown). If the previous instance was Claude and this instance is Codex (or vice versa), you read the same INBOX. **Do not** assume the previous instance had the same tool access you do — if a handoff says "run tool X," check whether X is available to you before acting.

For project-runtime interaction, prefer explicit task/result artifacts over raw
conversation continuity. A supervisor-to-project handoff should include:

- the live design intent or mental model when it materially affects the work
- `task_id`

    81	
    82	**Expanding the supervisor tick scope**: Rejected. The supervisor tick's safety

1. **Most dangerous assumption**

The design assumes the principal’s blanket delegation is sufficient to let an unattended model `commit` and `push` across every project, with Slack as the only post-hoc control surface (lines 15-18, 34-38, 56-57). If that assumption is wrong, the blast radius is workspace-wide: bad judgment lands on live branches in multiple repos before any human reviews it.

2. **Missing failure mode**

There is no explicit claim/ack/idempotency protocol for the handoff being executed. The tick “finds the oldest pending handoff” and runs it, but the ADR never says when that handoff is atomically marked in-progress, retired, or recognized as already partially executed after a crash, push rejection, or model timeout (lines 30-37, 64-67). In practice, this will cause duplicate execution, conflicting commits, or the same stale handoff being retried forever with slightly different mutations.

3. **Boundary most likely to be collapsed in practice**

The governance/execution boundary. The ADR says the supervisor tick must not collapse governance and code mutation into one session (lines 82-85), but this design effectively reintroduces that collapse through a different path: a supervisor-owned timer/template decides when unattended project code mutation happens, writes into the same handoff fabric, and relies on later executive review after code has already been pushed (lines 22-25, 34-38, 71-75). Under real use, “review later” becomes de facto approval, and the executive layer stops being a gate.

Overall verdict: this is operationally plausible but under-specified where it matters most: atomicity, rollback, and the real approval boundary. As written, it turns handoffs into executable authority without a durable control proving that execution happened once, safely, and on purpose.
